Hi, I’m new user of Tracks and I’m really happy with it.

Here it is my suggestion of improvement:

I find very useful the option “Show from” for an action, in order to defer an action to an specific date. But, could be possible to specify in “Show from” form something like “defer up to another action completion” ?
My reasoning: Sometimes an action precedes some other. So the second action should not be shown visible up to the previous one has been toggled as completed, and not at an predefined date.

I think this could be helpful if added, and in this way actions could be chained. In this way actions deferred after others won’t be shown before the action established as a prerequisite has been completed.

Masters, I will be waiting your feedback.


Thanks for so wonderful GTD application.

Hi, I’m new to Tracks too and I had the same idea. But when thinking it over I decided that this would be out of focus and contra productive. I think Tracks is not intended and should not be used as Project planning tool. The idea is to put everything you think you should do in, but not to organize it in the way you would organize project tasks.

If you start using it as project planning tool you soon will miss other things (at least I did). E. g. efforts, budgets, priorities, percentage of completion etc.  You also will think to much about actions before adding them. The correct way is: You think you should do something, add it to a default context (e.g. inbox), forget it, ‘till you review your inbox (daily, weekly, as you like it), then put it to some other context (next, waitiing for, sometime etc) and project or finish and close it (<2min).

If using it in project work you should think of a “Tracks project” as “Project Task” assigned to you. Thus it is much easier for you to add actions and trivial todos (like phone xyz, correct abc.cpp etc.) and to organize this actions in contexts.

To be honest, i had to read some sections of GTD again, before I got it and had a hard time to reorganize everything.

I agree with the original poster. Action chaining is a “must have” feature for me. If the Tracks team does not implement it, eventually I will do it for myself (and offer the code back, of course).

I previously wrote my own GTD app using Django and used it with great success for a couple of years. Tracks is much nicer and I want something I can recommend to others that I do not have to support (on a code level).

I implemented a sort of action chaining. For want of a better method, each action had a “sequence” field that was just a number to sort on. Like line numbers in Basic, you can always leave space for intermediate actions you forgot about, and give the same number for actions that do not depend on others.

Actions in the same project would not show in the Next Actions list if they followed other actions in the same project that were not done. It worked out extremely well because I did not distract myself with Next Actions that could not be carried out because of a “not done” preceding action.

And I did not spend undue time doing this. It was still very much the GTD quick process of inputs. But it did keep my Next Actions list effective, because I did not burden my thinking with actions that actually could not be “next”.

I don’t see action chaining as an abrogation of GTD. It models reality quite well. It doesn’t constitute or lead to using GTD/Tracks as a project manager like, say, Microsoft Project (ugh). Actually, GTD IS a project management system, just the definition of projects is different. And you are supposed to take time to think “enough” about a project and its actions, and THEN forget about it because the information is in a system you trust.

So congrats to Esfingie. Thanks for bringing this up. It is just what I want, too.

I think you should start out as simple as possible and take it from there (like the solution esfingie mentions). I.e. just add a many-to-one dependency relationship between todos (a todo can depend on only one other todo; there can be many todos depending on one ‘top’ todo). The gui suggestion of esfingie is nice to consider, i.e. use the show_from to either defer the action or let it depend on the completion of another…

@hboers: tracks is not intended as a project planning tool. But sometimes you added a next action, to find out that it is stalled because of some other next action it depends on. You can either remove the next action (because it is not a _next_ action anymore), put it in a hidden context (like putting it on someday/maybe) or perhaps a dependency mechanism like this could be used…  I don’t know what’s best, and it’s not ‘pure’ GTD, but there is a lot of people asking for this…

I disagree with some of this not being true to the spirit of GTD.  While I agree that GTD is not full project planning, the core of GTD is getting things out of your head into a trusted system.  I add a new project . . . how my mind works is to list out the tasks that need to be done.  Not in total detail, not with critical path, not people, etc - that is all full PM.  However, I need to dump all that I am thinking of and that ends up being a skeleton of a project plan.  I personally need the ability to order these and reorder them.  If you have that, then making them chained should be an easy option.
listing all of this is a critical aspect of GTD software, and right now Tracks is lacking that.  I will keep watching for sure, as I think it has huge potential and I want to use more open software.
